# Workspace spec — directory contract between freight-skills and your binding

The skills in `freight-skills` read company-specific state from a fixed workspace path on the VPS:

```
$HOME/.openclaw/workspace/shipping-rate-automation/
```

> Note: the workspace directory name `shipping-rate-automation` is a historical convention, deliberately decoupled from your company's repo name (which is `freight-<your-company>`). All skills assume this exact path — do not rename it.

Your private binding repo's `install.sh` is responsible for seeding this directory. The skills themselves only READ from it.

## Required files

| Path | Source | Required? | Purpose |
|---|---|---|---|
| `wecom/links.md` | Created by `freight-onboard` (or your binding's `workspace-seed/`) | **Required** | Truth source for DocID + sheet_id per smartsheet — see schema below |
| `raw/source-files/` | Operator drops files here manually | Required for scenario 2 | `运价表*.xlsx` + `运价信息*.docx` — read by `extract_rate_inputs.py` |
| `knowledge-base/` | Your binding's `workspace-seed/` | Optional | Company-specific knowledge for agent context |
| `scenarios/scenario-1-lead-profiling/` | Your binding's `workspace-seed/` | Optional | Per-company fixtures (customer leads template, batch result history) |
| `scenarios/scenario-2-daily-rate-promotion/runs/` | Created by cron at run time | Auto | Per-day output folders (housekeeping archives >30 day folders) |
| `scenarios/scenario-2-daily-rate-promotion/archive/` | Created by housekeeping step | Auto | tar.gz archives of old runs |

## `wecom/links.md` schema

Plain markdown with a strict shape. The skills parse it via `awk` / `grep` — keep the section headings and line prefixes intact.

```markdown
# WeCom workspace truth source — <Company Name>
# Generated by freight-onboard skill on <ISO timestamp>

## Scenario 1 workbench (拓客)
DocID: <scenario-1-docid>
- 客户线索表 sheet_id: <sheet_id>
- 待审核开发信 sheet_id: <sheet_id>

## Scenario 2 workbench (运价推广)
DocID: <scenario-2-docid>
- 运价表（人） sheet_id: <sheet_id>
- 运价信息（人） sheet_id: <sheet_id>
- 每日简报（AI） sheet_id: <sheet_id>
- 推广审核（AI+人） sheet_id: <sheet_id>
- 发送记录（AI） sheet_id: <sheet_id>
```

If you split the two scenarios across two separate WeCom workspaces (rather than one), keep one `## Scenario X workbench` block per workspace. The skill resolves each sheet independently.

## How skills read this file

In bash (rough pattern, real implementations may differ):

```bash
LINKS="$HOME/.openclaw/workspace/shipping-rate-automation/wecom/links.md"

# Scenario 2 DocID
S2_DOCID=$(awk '/^## Scenario 2/,0' "$LINKS" | awk -F': ' '/^DocID/ {print $2; exit}')

# A specific sheet_id under that scenario
RATE_TABLE_SHEET=$(awk '/^## Scenario 2/,0' "$LINKS" | awk -F': ' '/运价表/ {print $NF; exit}')
```

The skill SKILL.md doesn't prescribe the exact parsing implementation; that's an agent-side decision. The contract is just: read the labels in `links.md`, get the IDs back.

## Cron config

```
~/.openclaw/cron/jobs/<job-id>.json    (managed by `openclaw cron add`)
```

Generated by `freight-onboard/scripts/render-cron-config.py`. Schema:

```json
{
  "id": "<uuid>",
  "name": "<company-slug>-每日<HH:MM>运价简报与推广审核",
  "enabled": true,
  "schedule": { "kind": "cron", "expr": "MM HH * * *", "tz": "Asia/Shanghai" },
  "sessionTarget": "isolated",
  "wakeMode": "now",
  "payload": {
    "kind": "agentTurn",
    "timeoutSeconds": 900,
    "model": "openai-codex/gpt-5.5",
    "toolsAllow": ["exec", "read", "write"],
    "message": "<long agent prompt; see render-cron-config.py>"
  },
  "delivery": {
    "mode": "announce",
    "channel": "telegram|feishu|wecom|dingtalk",
    "to": "<chat ID or webhook URL>"
  }
}
```

## What lives outside the workspace

| Lives in | Path |
|---|---|
| Skill code | `~/.openclaw/extensions/freight-skills/` (managed by `openclaw plugins install`) |
| Skill symlinks for agent discovery | `~/.agents/skills/freight-{onboard,lead-profiling,rate-daily-promotion}/` |
| Cron job state | `~/.openclaw/cron/jobs/` |
| Secrets (bot tokens, mail passwords) | macOS Keychain / Linux secret-tool / 1Password / pass — NEVER in files |
| Customer data (real 客户线索表 rows) | WeCom only — never in git, never in workspace files |

If you `rm -rf ~/.openclaw/workspace/shipping-rate-automation/` you can re-run the binding's `install.sh` to rebuild it from `workspace-seed/`. The WeCom sheets themselves are unaffected — only the local cache is rebuilt.
